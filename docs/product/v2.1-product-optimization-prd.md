# JeriDevOps v2.1 产品优化 PRD

> 版本：v1.2  
> 日期：2026-05-06  
> 角色视角：高级产品经理  
> 依据：当前代码结构、V2 菜单、v2.0 Roadmap / Progress、ADR-0001~0008、前端核心页面抽样。

## 1. 背景

JeriDevOps 已经从 v1.x 的运维工具集合，演进到 v2.0 的 BizDevOps + GitOps 价值流平台。当前产品已经具备应用中心、CI/CD、Release 主单、ArgoCD GitOps、审批、数据库变更、事件时间线、日志统一台、告警中心、DORA、成本、安全、AI Copilot、Jira 规划源等能力。

但从用户视角看，产品仍存在“能力很全，但任务链路不够直”的问题。v2.0 的架构方向正确，v2.1 的重点不应继续横向堆模块，而应把现有能力压缩为少数高频工作流：应用接入、一次发布、风险处置、GitOps 迁移、持续改进。

## 2. 当前产品问题

### 2.1 信息架构仍偏重功能目录

V2 菜单已经按价值流重构，但仍有大量二级/三级入口。日志、告警、成本、安全、数据库、审批等能力分别有独立入口，用户需要理解平台结构后才能完成任务。部分菜单仍指向旧路径再由路由重定向，例如日志中心菜单指向 `/logs/center`，实际进入 `/logs/unified`，这会加重产品“历史功能合集”的感知。

### 2.2 首次成功交付路径不够产品化

应用详情已有仓库绑定、环境配置、交付流水线、发布单、GitOps 变更等能力，但缺少一个明确的“应用接入向导”。新用户不知道应按什么顺序完成：创建应用、绑定 Git 仓库、配置环境、绑定流水线、配置 ArgoCD、配置审批策略、跑通一次发布。

### 2.3 Release 主单是正确中枢，但体验还不够像“发布驾驶舱”

Release 详情已经包含审批、GitOps PR、发布、风险评分、变更项、事故关联，但页面仍像详情页叠加操作按钮。用户在关键时刻需要看到：当前处于生命周期哪一步、卡点是什么、谁需要处理、阻塞是否来自审批/发布窗口/风险/PR/Argo 同步/事故。

### 2.4 GitOps 中心与 Release 流程仍有割裂

ArgoCD 页面是强大的运维中心，Release 详情也能打开 PR，但发布负责人更关心“我的发布单是否已经生成 PR、是否合并、是否同步、是否漂移”。当前 GitOps 管理视角偏平台/SRE，Release 管理视角和 GitOps 状态之间还需要进一步融合。

### 2.5 工作台不够“个人行动中心”

Dashboard 已有 DORA、待办、快速入口和状态概览，但还没有形成按角色聚合的行动队列。开发、审批人、发布负责人、SRE、安全管理员看到的任务应该不同，并能直接处理：待审批、失败流水线、待合并 PR、生产漂移、待确认事故、高危漏洞、即将超预算资源。

### 2.6 可观测能力已有聚合，但 v2.1 暂不扩展事故处置

事件时间线、日志统一台、告警中心和 Incident 已存在，但一次事故处理应形成标准链路：告警触发 -> 关联发布/应用 -> 拉取日志/指标/Trace -> 建立事故 -> 执行缓解动作 -> 生成复盘 -> 推送 Confluence -> 回写 DORA。考虑 v2.1 交付周期，本版本不新增事故处置驾驶舱，仅保留现有 Incident 能力和 Release 关联能力，事故闭环放入后续版本。

### 2.7 治理能力散落，未前置到交付门禁

审批策略、发布窗口、部署锁、环境审核策略、安全扫描、成本建议、风险评分都已具备，但用户在发起发布时不能一眼看到“为什么可以发布/为什么不能发布”。治理应从后台配置变成交付过程中的 Gate Checklist。

### 2.8 Jira 权威源已确立，但需求到发布的追踪还不够自然

ADR-0007 确认 Jira 是规划权威源，Biz 模块已逐步只读化并支持 Jira 拉取。Jira Issue 到应用、流水线、Release、DORA 的链路仍是重要缺口，但用户已明确 Sprint 4 不需要，本期仅保留现状，不作为 v2.1 P0 交付范围。

### 2.9 GA 质量门槛仍是主要风险

进度文档显示 QA、E2E、性能回归、Pilot 应用切换、培训文档还未完全闭环。产品当前的问题不是缺少单点功能，而是缺少可验证的端到端成功标准。

## 3. 产品定位优化

### 3.1 一句话定位

JeriDevOps v2.1 是以应用为主体、以 Release 为变更中枢、以 GitOps 为发布事实源、以事件时间线为反馈闭环的企业级交付运营平台。

### 3.2 北极星指标

生产变更从“代码已构建”到“安全发布并完成反馈”的端到端周期时间。

### 3.3 核心原则

1. 任务优先，不让用户先理解模块。
2. 应用优先，所有交付、运行、成本、安全都能回到应用。
3. Release 优先，所有生产变更都通过 Release 主单沉淀。
4. GitOps 优先，线上状态以 Git + ArgoCD 为事实源。
5. 治理前置，审批、风险、安全、窗口、锁定都作为发布 Gate。
6. Copilot 嵌入流程，不作为独立聊天入口。

## 4. 用户角色

| 角色 | 主要目标 | 当前痛点 | v2.1 价值 |
|---|---|---|---|
| 开发工程师 | 快速接入应用并发起交付 | 不知道先配仓库、环境还是流水线 | 接入向导 + 一键生成交付链路 |
| 发布负责人 | 安全推动变更上线 | Release、审批、PR、Argo 状态分散 | Release 驾驶舱统一卡点 |
| 审批人 | 快速判断是否放行 | 风险上下文不足 | 审批页展示风险、变更项、窗口、历史事故 |
| SRE / 运维 | 处理漂移、告警、事故 | 日志/告警/事件需要跳转拼接 | GitOps 健康视图 + 现有事件时间线 |
| 安全/合规 | 阻断高风险发布并留痕 | 扫描与发布门禁割裂 | 安全 Gate + 审计证据 |
| 平台管理员 | 保障平台集成健康 | 不知道哪些应用未接入完整链路 | 数据健康与迁移中心 |
| 业务/PM | 追踪需求交付状态 | 需求与发布质量断链 | 后续补齐需求 -> Release -> DORA 追踪 |

## 5. 目标与成功指标

### 5.1 业务目标

1. 将平台从“工具集合”升级为“日常工作入口”。
2. 降低应用首次接入和首次 GitOps 发布成本。
3. 提升生产发布透明度和安全性。
4. 让 v2.0 GA 具备可验证质量。

### 5.2 量化指标

| 指标 | 当前基线 | v2.1 目标 |
|---|---:|---:|
| 新应用首次跑通交付链路耗时 | 未系统化 | <= 30 分钟 |
| Release 卡点可解释率 | 未系统化 | >= 95% 发布单展示明确阻塞原因 |
| GitOps Pilot 应用数 | 计划 3 个 | >= 10 个 |
| 发布详情首屏加载 | 需回归 | <= 1.5s |
| 发布单 PR 生成成功率 | 需统计 | >= 95% |
| 核心流程 E2E | 0 未启动 | >= 22 条全绿 |

## 6. 产品方案

### 6.1 全局行动中心

将 Dashboard 从“概览页”升级为“我的行动中心”。

P0 能力：

- 按角色聚合任务：我的审批、我的发布、失败流水线、待合并 GitOps PR、漂移应用、高危安全风险。
- 每个任务展示状态、影响范围、处理人、截止时间、下一步动作。
- 支持“交付优先 / 运行优先 / 平台优先”视图，但数据源必须来自真实业务对象。
- 支持从任务直接跳转到对应 Release、Approval、Pipeline、ArgoCD、Security 页面。

验收标准：

- 登录后 3 秒内可看到至少 5 类行动卡片。
- 所有行动卡片必须有明确处理按钮，不能只做展示。
- 权限不足的任务不展示操作按钮，只展示只读链接或不展示。

### 6.2 应用接入向导

在应用中心新增“接入向导”，面向新应用或链路不完整应用。

P0 步骤：

1. 创建/选择应用。
2. 绑定标准 Git 仓库。
3. 配置环境：dev/test/staging/prod、namespace、deployment、默认分支。
4. 绑定或创建交付流水线。
5. 绑定 GitOps 仓库和 ArgoCD 应用。
6. 配置发布策略、审批策略、发布窗口。
7. 跑通一次测试环境发布。

P1 能力：

- 接入完成度评分。
- 缺失项自动生成待办。
- 支持从已有应用补齐配置。

验收标准：

- 新应用可通过向导跑通一条 dev/test 发布。
- 应用详情展示“交付链路完整度”，低于 80% 时给出缺失项。

### 6.3 Release 发布驾驶舱

改造 Release 详情为生命周期驱动页面。

P0 生命周期：

草稿 -> 风险评估 -> 提交审批 -> 审批通过 -> GitOps PR 打开 -> PR 合并 -> ArgoCD 同步 -> 发布完成 -> 反馈观察。

P0 Gate Checklist：

- 变更项完整性。
- 审批状态。
- 发布窗口。
- 部署锁。
- 风险评分。
- 安全扫描。
- 数据库工单。
- Nacos 配置变更。
- GitOps PR 状态。
- ArgoCD Sync / Health / Drift。

P0 页面能力：

- 顶部展示当前阶段、阻塞原因和下一步动作。
- 右侧风险卡展示风险命中规则、影响环境、建议处理。
- 变更项支持关联 Pipeline Run、Nacos Release、DB Ticket。
- GitOps 状态直接内联，不再要求跳到 ArgoCD 页判断。
- 发布完成后进入观察期，展示当前 Release 的 GitOps 同步、健康和基础运行状态。

验收标准：

- 任一非终态 Release 都能明确展示“下一步由谁处理”。
- Gate 未通过时主按钮禁用并展示原因。
- PR / Argo 同步状态变化后 Release 页面可刷新同步。

### 6.4 GitOps 迁移与健康中心

保留 ArgoCD 管理能力，但新增面向平台管理员的迁移视图。

P0 能力：

- 应用 GitOps 接入状态：未接入、部分接入、已接入、漂移中、同步异常。
- Pilot 应用看板：接入进度、发布成功率、DORA 变化、失败原因。
- 漂移应用队列：按环境、应用、严重度排序。
- 一键生成 GitOps 修复建议，必要时生成变更请求。

验收标准：

- 平台管理员能筛出所有未接入 GitOps 的生产应用。
- 每个漂移应用展示 drift 来源、最近 Release、建议动作。

### 6.5 统一治理 Gate 中心

将审批、发布窗口、部署锁、风险、安全、成本、数据库/Nacos 变更统一为“环境 Gate”。

P0 能力：

- 环境维度配置 Gate 模板：dev/test/staging/prod。
- Release 详情读取 Gate 结果并展示。
- Gate 结果入审计日志。
- 支持强阻断、弱提醒、人工豁免三类策略。

验收标准：

- 生产环境发布可配置“严重漏洞强阻断”“窗口外强阻断”“DB 工单未完成强阻断”。
- 所有豁免必须记录原因、审批人和有效期。

### 6.6 Jira 到交付追踪（本期不做）

用户已明确 Sprint 4 不需要，因此 Jira 到 Release / DORA 的追踪能力从 v2.1 P0 移出，保留为后续版本候选项。

后续候选能力：

- 在 Jira Issue 拉取列表、Biz Requirement 详情、Release 详情展示互链。
- Release 新建时可选择 Jira Issue / FixVersion。
- DORA 分析支持按 Jira Project / Version 过滤。

后续验收标准：

- 一个 Jira Issue 可以看到关联应用、Release、状态和上线时间。
- 一个 Release 可以看到关联 Jira Issue 和业务版本。

### 6.7 Copilot 流程化动作（后续版本）

Copilot 不新增菜单。v2.1 仅保留现有全局 Copilot 和上下文能力，不新增流程化动作、草稿确认接口和动作审计表。后续版本可在 Release、Incident、Application、Logs 页面补充流程化动作。

后续候选动作：

- Release：解释风险、生成发布说明、生成回滚计划。
- Incident：生成排查清单、总结时间线、生成 Postmortem 草稿。
- Application：检查接入缺失项。
- Logs：总结异常日志。

后续版本验收约束：

- Copilot 所有写操作必须先生成草稿，由用户确认后执行。
- 所有 Copilot 动作写入审计。

## 7. 范围

### 7.1 v2.1 P0 范围

- 全局行动中心。
- 应用接入向导 MVP。
- Release 发布驾驶舱。
- GitOps 接入健康视图。
- 统一 Gate 结果模型和 Release 展示。
- 22 条核心 E2E 和 Pilot 应用切换。

### 7.2 v2.1 不做

- 不新增新的 CD 引擎，继续以 ArgoCD GitOps 为唯一发布实现。
- 不恢复 Jenkins/K8s 直发路径。
- 不新增被 ADR-0005 移除的通知渠道。
- 不把 Copilot 做成独立菜单或独立机器人产品。
- 不新增事故处置驾驶舱、Incident 处置动作表或 Copilot 流程化动作。
- 不做 Sprint 4 范围：Jira Issue / FixVersion 互链、DORA Jira Project / Version 过滤、GA 质量硬化专项。
- 不重做 UI 技术栈。

## 9. 数据与接口建议

- `ActionItemDTO`：行动中心统一任务。
- `ApplicationReadinessDTO`：应用交付链路完整度。

- `application_readiness_checks`：应用接入检查结果。

## 10. 验收用例

1. 新用户创建应用，完成仓库、环境、流水线、GitOps 绑定，并在 test 环境跑通一次发布。
2. 发布负责人从工作台进入一个阻塞 Release，看到阻塞原因是“窗口外 + 高危漏洞”，并处理或申请豁免。
3. 审批人打开审批详情，看到风险评分、变更项、最近事故，完成审批。
4. PR 合并后，Release 页面显示 ArgoCD 同步状态和健康状态。
5. 平台管理员查看 GitOps 迁移健康，筛出未接入生产应用并生成待办。

## 11. 风险与应对

| 风险 | 等级 | 应对 |
|---|---|---|
| 端到端链路跨模块，开发联调复杂 | 高 | 先定义统一 DTO 和只读聚合接口，再做写操作 |
| 现有页面较大，改造容易回归 | 高 | 优先新增子组件和聚合 API，避免大范围重写 |
| Gate 规则阻断影响真实发布 | 高 | 默认观察模式，生产强阻断需管理员显式开启 |
| GitOps Pilot 应用不足 | 中 | 提供迁移中心和应用接入完整度评分 |
| E2E 覆盖不足 | 高 | P0 需求必须绑定 E2E 场景，未通过不得 GA |
