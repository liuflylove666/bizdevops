# JeriDevOps v2.1 开发计划

> 版本：v1.2  
> 日期：2026-05-06  
> 关联 PRD：`docs/product/v2.1-product-optimization-prd.md`  
> 目标周期：6 周 / 3 个 Sprint  
> 目标：把 v2.0 已完成的模块能力收敛为端到端可验证的交付闭环。

## 1. 交付目标

v2.1 不以新增模块数量为核心目标，而以“用户能否顺畅完成关键任务”为交付标准：

1. 新应用 30 分钟内完成接入并跑通 test 环境发布。
2. Release 页面能解释所有发布卡点，并给出下一步动作。
3. GitOps 接入、同步、漂移、PR 状态对发布负责人可见。
4. 至少 10 个 Pilot 应用切到 GitOps。
5. 核心 E2E 覆盖 Sprint 1-3 的 P0 闭环。

## 4.1 当前开发状态（2026-05-06）

已进入开发并落地 Release Gate 主链路：

- 后端新增 `ReleaseGateService`，覆盖变更项、审批、发布窗口、部署锁、风险、安全扫描、数据库工单、Nacos、GitOps PR、ArgoCD Sync/Health/Drift。
- 新增 API：`GET /releases/:id/gates`、`POST /releases/:id/gates/refresh`，并接入 v1/v2 Release 路由。
- 发布动作前执行 Gate 评估；强阻断未通过时禁止发布。
- 新增 `release_gate_results` 快照表、模型、DTO、迁移脚本。
- 前端 `ReleaseDetail.vue` 增加 Gate Checklist，Gate 阻断时禁用发布按钮并展示下一步动作。
- 已补充 `GateService` 单测，覆盖草稿阻断、GitOps PR 阻断、安全扫描通过、ArgoCD 未同步阻断、快照持久化。

尚未完成：

- 应用接入向导、Application Readiness、Workspace Action Center。
- GitOps 迁移健康中心。
- Gate 模板配置、人工豁免审计、审批详情页 Gate 摘要。

## 5. Sprint 计划

### Sprint 1：行动中心与应用 Readiness（W1-W2）

目标：让用户登录后知道该做什么，让应用详情知道缺什么。

后端：

- 新增 `workspace/actions` 聚合服务，汇总审批、Release、Pipeline Run、GitOps ChangeRequest、安全风险。
- 新增 `ApplicationReadinessService`，检查仓库、环境、流水线、GitOps、审批策略、发布窗口。
- 新增 DTO：`ActionItemDTO`、`ApplicationReadinessDTO`。
- 新增 API：`GET /workspace/actions`、`GET /applications/:id/readiness`、`POST /applications/:id/readiness/run`。

前端：

- 改造 `web/src/views/Dashboard.vue`，新增“我的行动”队列。
- 改造 `ApplicationList.vue`，展示接入完整度和缺失项。
- 改造 `ApplicationDetail.vue`，新增接入完整度卡片。

QA：

- E2E 1-6：登录、行动队列、应用完整度、权限过滤、跳转、空状态。

验收：

- 至少 5 类行动卡片接入真实数据。
- 一个链路不完整应用能列出明确缺失项。

### Sprint 2：应用接入向导与 Release 驾驶舱骨架（W3-W4）

目标：跑通新应用接入和发布主单生命周期视图。

后端：

- 新增应用接入向导保存接口，复用现有应用、仓库、环境、流水线、ArgoCD 能力。
- 新增 `ReleaseTimelineService` 聚合 Release 阶段。
- 新增 `ReleaseGateService` MVP，先只读聚合审批、窗口、锁、风险、PR、Argo 状态。
- 新增 API：`GET /releases/:id/timeline`、`GET /releases/:id/gates`、`POST /releases/:id/gates/refresh`。

前端：

- 新增应用接入向导组件，嵌入应用列表和应用详情。
- 改造 `ReleaseDetail.vue` 顶部为生命周期进度条。
- 新增 Gate Checklist 组件。
- GitOps PR / ArgoCD 状态在 Release 页内联展示。

QA：

- E2E 7-14：新应用接入、仓库绑定、环境配置、流水线绑定、Release 生命周期、Gate 阻塞、PR 状态刷新、Argo 状态刷新。

验收：

- 新应用可通过向导完成 test 环境交付链路配置。
- Release 页面能展示当前阶段、下一步动作、阻塞原因。

### Sprint 3：GitOps 迁移健康与 Gate 强化（W5-W6）

目标：平台管理员能推动 Pilot 扩容，生产发布有可解释门禁。

后端：

- 新增 `GitOpsReadinessService`，聚合应用 GitOps 接入状态、漂移、同步健康。
- `ReleaseGateService` 增加安全扫描、DB 工单、Nacos 变更、环境审核策略。
- 新增 Gate 快照表 `release_gate_results`。
- 支持 Gate 策略的强阻断、弱提醒、人工豁免。

前端：

- 改造 `ArgoCDIntegration.vue`，新增“迁移健康”Tab。
- 改造 Release Gate Checklist，支持强阻断/提醒/豁免展示。
- 审批详情页展示 Release Gate 摘要。

SRE：

- 选择并接入 10 个 Pilot 应用。
- 建立 GitOps 迁移问题清单。

QA：

- E2E 15-21：GitOps 健康列表、漂移处理、Gate 阻断、Gate 豁免、审批上下文、Pilot 发布、审计记录。

验收：

- 10 个 Pilot 应用在迁移健康中心可追踪。
- 生产发布 Gate 未通过时不能误操作发布。

## 9. 发布策略

1. Sprint 1-2 在开发环境开启，不进入生产强阻断。
2. Sprint 3 开始选择 10 个 Pilot 应用灰度。
3. Gate 强阻断默认关闭，仅在 Pilot 应用生产环境开启。
4. Sprint 3 验收和培训后开启全量菜单入口。
5. 旧路径继续保留重定向 30 天，但菜单不再暴露旧入口。

## 11. 关键风险

| 风险 | 等级 | 缓解方案 |
|---|---|---|
| Release 页面改造影响核心发布 | 高 | 新增组件渐进替换，保留原 API 和操作逻辑 |
| Gate 强阻断误伤发布 | 高 | 默认观察模式，强阻断只对 Pilot 应用开启 |
| Readiness 检查口径争议 | 中 | 先做透明规则展示，允许管理员配置权重 |
| ArgoCD 实例状态不稳定 | 中 | 状态缓存 + 手动刷新 + 错误可解释 |
| E2E 环境依赖复杂 | 高 | Sprint 1 即搭建稳定测试数据和 Mock 外部依赖 |
